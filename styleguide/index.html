iOS Type Scale

Apple's Dynamic Type scale at the default Large size. Display styles (20pt+) use SF Pro Display; text styles (19pt and below) use SF Pro Text. The base size is 17px (Body & Headline) — Apple's default for comfortable one-handed reading at typical phone viewing distance (~25–30 cm). It aligns with UIFont.preferredFont at the "Large" Dynamic Type setting, ensuring text is legible without zooming while leaving room for the scale to grow in accessibility sizes.

Usage Notes

Optical sizing

SF Pro automatically adjusts letter shapes based on size via the opsz axis. At small sizes, letters are wider with more open counters for legibility. At large sizes, spacing tightens for elegance. Prefer SF Pro over explicit Display/Text cuts to let the browser handle this.

Tracking matters

Apple's tracking values are carefully tuned per style. Display sizes use positive tracking (slightly open), while text sizes use negative tracking (slightly tight). Always apply the tracking token alongside size and weight.

Width variants for maps

Compressed and Condensed widths are especially useful for map labels, data-dense panels, and compact navigation where horizontal space is limited. Use font-stretch or the named CSS variable to access them.

Rounded for warmth

SF Pro Rounded shares identical metrics with SF Pro but has softer stroke terminals. Use it for badges, tags, onboarding text, or any moment where the UI should feel approachable. Avoid mixing Rounded and default in the same text block.

Platform availability

SF Pro is pre-installed on macOS, iOS, iPadOS, watchOS, and tvOS. It is not available on Windows or Android. The CSS fallback stack (-apple-system, BlinkMacSystemFont, system-ui, sans-serif) ensures graceful degradation to each platform's system font.

Overview

The Livemap Design System is the shared language for everyone building Livemap. It defines the typography, color, spacing, and component decisions that keep the product coherent across iOS, Android, and the web.

This styleguide is the single source of truth. Every token, type scale, and pattern documented here is what ships in the product. If it's not in the system, it shouldn't be in the build. Use it to design with real values, review work against a known baseline, and onboard new team members without tribal knowledge.

Design Principles

Four principles guide every design decision at Livemap. When you're unsure which direction to take, come back to these.

01

Map-first

The map is the product. UI should support spatial understanding, never compete with it. Keep chrome minimal, let geography do the talking.

02

Clarity over cleverness

Every element earns its place. Prefer clear labels over icons, visible states over hidden ones, and obvious actions over discoverable ones.

03

Platform-native feel

SF Pro on Apple, Roboto on Android, system fonts on the web. Respect each platform's conventions for navigation, gestures, and density.

04

Accessible by default

Contrast ratios, touch targets, and semantic markup aren't afterthoughts. Build for the widest range of people and contexts from the start.

Use the Pencil kit or this styleguide as your starting point. Pull colors, type styles, and spacing from the system — don't eyeball values.

2

Validate across platforms

Use the platform toggle (HTML / iOS / Android) in this styleguide to verify that your design maps cleanly to each target. Naming conventions differ — the system handles the translation.

3

Document what you ship

If you introduce a new token, pattern, or component, add it to the styleguide before closing the ticket. Undocumented additions create drift.

4

Review against the system

During design review, check that new work uses existing tokens and patterns. If something doesn't fit, discuss whether to extend the system or adjust the design.

Contributing

The design system is a living product. Here's how to contribute well.

Check existing patterns first. Before proposing something new, verify it doesn't already exist in the system. Duplication is the most common source of drift.
Propose in context. When you need a new token or pattern, share the specific use case. "I need a 14px muted label for map overlays" is better than "can we add another gray?"
Update the styleguide. Every addition or change should be reflected here. Add the section, write the usage note, show the example. If it's not documented, it's not in the system.
Think cross-platform. A token that only works on iOS isn't a system token. Consider how it maps to Android and web before committing.
Keep it tight. Resist the urge to add options. A design system with 6 grays and 4 weights is more useful than one with 12 of each.

Contrast & Accessibility

WCAG 2.1 AA requires a minimum contrast ratio of 4.5:1 for normal text and 3:1 for large text (≥18px bold or ≥24px regular). All primary and secondary label colors pass.

Label on White The quick brown fox jumps over the lazy dog 16.75:1 · AA Pass AAA Pass
Secondary on White The quick brown fox jumps over the lazy dog 4.97:1 · AA Pass
Tertiary on White The quick brown fox jumps over the lazy dog 1.87:1 · AA Fail
Label on Surface Secondary The quick brown fox jumps over the lazy dog 16.34:1 · AA Pass AAA Pass
Rule: Never use Tertiary or Quaternary label colors for text that must be readable. These colors are for decorative or supplemental elements only — watermarks, subtle dividers, background patterns. All readable text must use Label or Secondary.

Usage Guidelines

Follow these rules to keep icons readable, accessible, and consistent across all platforms.

Do

Use Lucide icons consistently. Pick one icon per concept and stick with it across the entire product.

Don't

Don't use filled icons, mix stroke weights, or change line cap/join styles. Avoid non-Lucide icons.

Do

Color icons with semantic tokens: --color-label for primary, --color-gray-2 for secondary, or a tint for interactive states.

Don't

Don't hardcode colors. Always use design tokens so icons adapt to light/dark mode and maintain contrast ratios.

Do

Delete

Pair icons with text labels for primary actions. Icons alone are acceptable only for universal patterns (close, back, search).

Don't

Delete this item now

Don't shrink icons below 16px or pair a tiny icon with a long label. Keep the icon visually balanced with adjacent text.

Accessibility: All interactive icons must have an aria-label or accompanying visible text. Decorative icons should use aria-hidden="true". Ensure a minimum 44×44px touch target around tappable icons per Apple HIG, using padding when the icon itself is smaller.

Terrain & Surfaces

Base map style principles: minimal chrome, muted terrain, content-forward design.

Design direction

  • Principles: minimal chrome, muted terrain, content-forward
  • Building footprints: subtle fill, no stroke at low zoom
  • Water rendering: flat fill, no wave texture
  • Park fills: desaturated green, no pattern overlay

Map style configuration is provider-specific
