iOS Type Scale

Apple's Dynamic Type scale at the default Large size. Display styles (20pt+) use SF Pro Display; text styles (19pt and below) use SF Pro Text. Usage Notes

Optical sizing

SF Pro automatically adjusts letter shapes based on size via the opsz axis. At small sizes, letters are wider with more open counters for legibility. At large sizes, spacing tightens for elegance. Prefer SF Pro over explicit Display/Text cuts to let the browser handle this.

Tracking matters

Apple's tracking values are carefully tuned per style. Display sizes use positive tracking (slightly open), while text sizes use negative tracking (slightly tight). Always apply the tracking token alongside size and weight.

Width variants for maps

Compressed and Condensed widths are especially useful for map labels, data-dense panels, and compact navigation where horizontal space is limited. Use font-stretch or the named CSS variable to access them.

Rounded for warmth

SF Pro Rounded shares identical metrics with SF Pro but has softer stroke terminals. Use it for badges, tags, onboarding text, or any moment where the UI should feel approachable. Avoid mixing Rounded and default in the same text block.

Platform availability

SF Pro is pre-installed on macOS, iOS, iPadOS, watchOS, and tvOS. On Android, Livemap uses Google Sans. The CSS fallback stack (-apple-system, BlinkMacSystemFont, system-ui, sans-serif) ensures graceful degradation to each platform's system font.

Overview

      

        The Livemap Design System is the shared language for everyone building Livemap. It defines the typography, color, spacing, and component decisions that keep the product coherent across iOS, Android, and the web.

Design Principles

      

        Four principles guide every design decision at Livemap. When you're unsure which direction to take, come back to these.
      


      

        

          
01

          
Map-first

          
The map is the product. UI should support spatial understanding, never compete with it. Keep chrome minimal, let geography do the talking.

        

        

          
02

          
Clarity over cleverness

          
Every element earns its place. Prefer clear labels over icons, visible states over hidden ones, and obvious actions over discoverable ones.

        

        

          
03

          
Platform-native feel

          
SF Pro on Apple, Google Sans on Android, system fonts on the web. Respect each platform's conventions for navigation, gestures, and density.

        

        

          
04

          
Accessible by default

          
Contrast ratios, touch targets, and semantic markup aren't afterthoughts. Build for the widest range of people and contexts from the start.

Contributing

      

        The design system is a living product. Here's how to contribute well.
      


      

        

          

          
Check existing patterns first. Before proposing something new, verify it doesn't already exist in the system. Duplication is the most common source of drift.

        

        

          

          
Propose in context. When you need a new token or pattern, share the specific use case. "I need a 14px muted label for map overlays" is better than "can we add another gray?"

        

        

          

          
Update the styleguide. Every addition or change should be reflected here. Add the section, write the usage note, show the example. If it's not documented, it's not in the system.

        

        

          

          
Think cross-platform. A token that only works on iOS isn't a system token. Consider how it maps to Android and web before committing.

        

        

          

          
Keep it tight. Resist the urge to add options. A design system with 6 grays and 4 weights is more useful than one with 12 of each.

Animation Principles

      

        Follow these principles to keep motion purposeful and consistent across all platforms.
      


      

        1. Purposeful: Every animation should serve a function — guiding attention, confirming an action, or showing spatial relationships. Remove motion that exists only for decoration.
      

      

        2. Quick: Prefer shorter durations. Users should never feel like they're waiting for an animation to finish. If a transition takes longer than 500ms, reconsider whether it's needed.
      

      

        3. Natural: Use easing curves that mimic physical movement. Elements should accelerate and decelerate smoothly — never move at a constant speed (linear).
      

      

        4. Consistent: Use the same easing and duration for the same type of transition everywhere. A dropdown should feel identical whether it's in the header, sidebar, or a form.
      

      

        5. Respectful: Honor prefers-reduced-motion. When the user requests reduced motion, disable transitions or replace them with instant cuts. Never remove state changes — only remove the animation.

Core Principle

            

              The map describes the environment. It never judges places, predicts threats, or labels areas as dangerous. It shows what is there — lighting, infrastructure, open venues, paths — and lets the user decide.
            

            

              Description over Judgment. The map is the product. What appears on it — and what does not — defines the user’s experience. Every element must be observable, environmental, descriptive, and stable or slow-changing.
            

            

              The test: “If I were standing there, could I reasonably notice this?” If an element would make someone stop walking just by seeing the map, it does not belong on the map.
            

          


          
          

            
A. Physical Environment (Core Layer)

            
Always allowed. These describe the city as it is — the baseline reality layer. They are observable and permanent.


            

              

                
🛤

                
Streets, paths, crossingsThe navigable network as it physically exists.

              

              

                
🌉

                
Bridges, tunnels, underpassesVertical transitions and grade-separated crossings.

              

              

                
🌳

                
Parks, trees, open spacesGreen spaces, plazas, and pedestrian areas.

              

              

                
💧

                
Water, terrain, slopeRivers, lakes, elevation changes. Geography as context.

              

              

                
🏛

                
Buildings as landmarksFor orientation, not ratings. No value judgement attached.

              

            

          


          
          

            
B. Infrastructure Characteristics

            
Allowed when shown neutrally. These are critical for the product — they describe how the environment is built.

            

              Rule: The map may show how something is built, not how dangerous it is.
            


            

              

                
💡

                
Lighting presenceEven / uneven illumination. From municipal data, shown as gradient, not binary.

              

              

                
🚲

🛣

🚧

What Must Stay Off the Map

            
Non-negotiable. The following must never appear on the map, even subtly. The map shows conditions, not decisions.


            

              
Fear Signals

              

                
Never showWhy

                
                  
Red zones or colour-coded “danger” areasThey create panic, stigmatise places, and override user judgement.

                  
Heatmaps labelled as dangerTurns data visualisation into a fear instrument.

                  
Warning triangles, exclamation marks, sirens, shieldsAlarm iconography designed to trigger fight-or-flight.

Never showWhy

                
                  
“Unsafe street” / “Avoid this area”Judgement, not description. Decisions belong in route comparison.

                  
“High crime area”Crime data correlates with policing patterns, not actual risk.

                  
“Safer route” badges on the mapComparative language belongs in route comparison UI, not pinned to geography.

                  
Risk scores or safety ratingsReduces complex environments to a single number. False objectivity.

                
              

            


            

              
Predictive Assumptions

              

                
Never showWhy

                
                  
“Likely dangerous” / “Risk increases here”Speculation framed as fact. Creates anxiety about unknowable states.

                  
“You may feel unsafe”Emotional inference. No behavioural assumption belongs on the map.

                  
Incident markers or report pinsPast incidents say nothing about current state. Turns the map into a fear register.

                
              

            


            

              
Social Profiling

              

                
Never encodeWhy

                
                  
Demographics of an areaThis is how bias sneaks in. People are not map data.

                  
“Busy vs quiet” framed as good/badPresence of people is not inherently positive or negative.

                  
Crowd density or “people nearby”Surveillance signal. Privacy violation for those being counted.

                  
User-submitted “feeling unsafe” reportsSubjective, biased, unverifiable. Disproportionately targets marginalised communities.

                
              

            

          


          
          

            
Where “Safety” Belongs Instead

            
Safety is not absent from the product — it is expressed through language, choices, and comparison, never through map markers. The map shows conditions. The UI helps the user interpret them.


            

              

                
Route comparison views

                
“Route A has more consistent lighting” — not a glowing red/green map. Side-by-side comparison of environmental characteristics, presented as facts.

              

              

                
Explanatory panels

                
Tapping a route segment can reveal details: “This section has uneven lighting” or “Protected bike lane for 400m”. Information on demand, not overlaid on geography.

              

              

                
User-controlled modes

                
The user chooses what to see. Overlays are opt-in. Preferences persist. No overlay is forced on the user by default.

              

            

          


          
          

            
Overlay Rules

            
Overlays are powerful — and dangerous if unmanaged. Each overlay answers exactly one question and must follow strict rules.


            

              

                
One question per overlay

                
Each overlay answers exactly one question: “Where is lighting present?” or “Where do bike lanes exist?” Never combine multiple meanings into one overlay.

              

              

                
Overlays can co-exist

                
Multiple overlays may be active in parallel. However, a maximum of 2 overlays may be visible at once to prevent cognitive overload.

              

              

                
Third overlay handling

                
If the user selects a third overlay: prompt to replace one, or auto-disable the oldest with a brief explanation. Never silently stack three layers.

              

              

                
Emergency and status signals

                
Never appear on the map. They belong in UI chrome (top bar, bottom sheet, notification), not pinned to geography.

              

            

          


          
          

            
Overlay Priority & Stacking

            
Layers stack in a fixed priority order. Higher layers always render above lower ones.


            

              

                
1

                
Base map (streets, buildings, water, terrain)

                
Always visible

              

              

                
2

                
Active route path

                
Always visible during navigation

              

              

                
3

                
Infrastructure overlays (lighting, lanes)

                
Max 2 active · user-toggled

              

              

                
4

                
Context overlays (weather, time-based)

                
Optional · dismissible · shares max-2 limit

              

              

                
5

                
User position + direction

                
Always visible

              

            


            

              Stacking rule: Lower-priority layers must never visually compete with higher-priority ones. If a conflict occurs, the lower layer reduces opacity or defers. Emergency or status overlays belong in UI chrome, never on the map.
            

          


          
          

            
Overlay Visual Behaviour

            
How overlays appear, disappear, and coexist. All transitions follow the Motion principles.


            

              

                
                  
BehaviourRule

                
                
                  
AppearingFade in over 300ms. No slide, bounce, or scale. The overlay materialises in place.

                  
DisappearingFade out over 200ms. Faster than appearing — removal should feel instant.

                  
SwitchingCross-fade over 300ms. The outgoing overlay fades while the incoming one appears. No gap, no double-stack.

                  
Zoom responseMarkers cluster at far zoom, expand at close zoom. Transitions are instant — no animated regrouping.

                  
Pan / moveExpanded marker details auto-collapse on pan. The map movement is the user’s intent — overlays must not compete.

                
              

            

          


          
          

            
Visual Density


            

              

                
Maximum marker density

                
No more than 12 individual markers visible at any zoom level. Beyond that, cluster. A cluttered map is a useless map — especially at night, on a small screen, while moving.

              

              

                
Overlay opacity

                
Informational overlays render at 40–60% opacity so the base map remains legible underneath. The overlay informs, it does not obscure.

              

              

                
Label collision

                
When labels overlap, lower-priority labels hide entirely. No partial overlap, no truncation. A label is either fully readable or not shown at all.

              

              

                
Colour restraint

                
The map uses the neutral palette. Colour is reserved for the route path and active-state indicators. Everything else is monochrome. The map should feel calm, not like a dashboard.

              

            

          


          
          

            
Rationale

            
This approach ensures the product stays true to its philosophy and clearly differentiates Livemap from crime heatmap tools, “safe city” dashboards, and surveillance-based navigation. We are building orientation, not surveillance.

Overview

      

        The Flare system handles contextual, serendipitous notifications in Livemap. Flares appear when something interesting is nearby — a friend, an event, a discovery. They are gentle, never blocking, and always dismissible.
        Flares create moments of discovery, not obligation.
      


      

        Core philosophy: The Flare is not an alert. It is a suggestion that something interesting exists nearby. It must feel like a happy accident, not a push notification. The user should smile, not startle.

Principles

      

        Every Flare design decision must follow these rules. The system must choose the best moment, not every moment.
      


      

        
✨

        

          
Serendipity, not notification

          
The Flare is a gentle suggestion that something interesting exists nearby. It must feel like a happy accident, not a push notification.

        

      

      

        
👋

        

          
Always dismissible, never blocking

          
The Flare must never obstruct navigation or critical UI. It appears at the top, above content, and can be swiped away or ignored. Auto-dismisses after 12 seconds if not interacted with.

        

      

      

        
⏱

        

          
Frequency discipline

          
Maximum one Flare per session or per 30 minutes, whichever is longer. Multiple flares in succession feel like spam. The system must choose the best moment.

        

      

      

        
🌍

        

          
Proximity-driven

          
Flares only fire based on real proximity — friends nearby, events within walking distance, discoveries on the current route. No speculative or time-based triggers.

        

      

      

        
🛡

        

          
Respect privacy

          
Location sharing is opt-in per friend. No one appears on another user's Flare without explicit consent. Users can pause Flare entirely in settings.

Spacing

            
Live Activities follow a concentric design principle — rounded shapes must nest inside each other with even margins. Content never touches the Dynamic Island walls.


            

              
Concentric Design Rule

              
Blur your content shape and verify the result is concentric with the Dynamic Island border. Inner corner radii should be: outer radius − padding. For the 44 pt expanded island, 20 pt inner padding gives 24 pt inner radii.

Typography in Dynamic Island

              
Compact views: 15 pt text, 22 pt line height. Use heavy-weight, large text for readability at a glance. Icons/images: 24 pt bounding box in compact views.

Do

                
Use bold, high-contrast colors against the black island. Tint icons and key values with system blue or your brand accent.

              

              

                
Don't

                
Don't use dark grays or muted tones — they disappear against the black background. Never set a custom Dynamic Island background color.

              

            


            

              
Lock Screen background

              
Lock Screen widgets can use .activityBackgroundTint() with opacity. Apply sparingly — users customize wallpapers, so semi-transparent backgrounds work best. The system auto-generates a dismiss button from your foreground and background colors.

            


            

              
Vibrancy levels

              
iOS defines vibrancy levels for labels, fills, and separators: default > secondary > tertiary > quaternary. Use secondary vibrancy for supporting text and quaternary for subtle separators.

Accessibility

            
Every Livemap surface must meet WCAG 2.1 AA. Follow these checks for every screen and component.


            

              

                
                All text meets 4.5 : 1 contrast against its background (3 : 1 for large text ≥ 18pt bold / 24pt regular).
              

              

                
                Interactive elements have a minimum touch target of 44 × 44 px (var(--touch-target-minimum)).
              

              

                
                No text is set below 11 px (var(--font-size-minimum)). Tappable text uses ≥ 13 px (var(--font-size-minimum-interactive)).
              

              

                
                Color is never the sole indicator of state — always pair with an icon, label, or shape change.
              

              

                
                Focus rings are visible on all interactive elements for keyboard navigation.
              

              

                
                Images and icons include descriptive alt text or aria-label; decorative images use alt="".
              

              

                
                Motion respects prefers-reduced-motion — disable non-essential animations when set.

Responsive Design

            
Livemap uses a mobile-first approach with three breakpoints. Content reflows, but spacing tokens remain consistent across sizes.


            

              

                

                  
                

                

                  
Compact — up to 390 px

                  
Single-column layout. Full-bleed cards. Bottom sheet navigation. Touch targets at maximum size.

                

              

              

                

                  
                

                

                  
Medium — 391 px to 768 px

                  
Two-column grid for cards. Side-by-side form fields. Collapsible sidebar available but hidden by default.

                

              

              

                

                  
                

                

                  
Expanded — 769 px and above

                  
Full sidebar navigation. Three-column card grids. Generous whitespace with --space-32 gutters. Max content width: 1040 px.

                

              

            

          


          
          

            
Voice & Tone

            
Livemap speaks clearly and confidently. The tone shifts with context — instructional in guides, encouraging in empty states, concise in labels. The core rule: describe, never judge.


            

              

                
Do

                
Use short, direct sentences. Lead with the action. “Tap the pin to see details.”

              

              

                
Don't

                
Avoid passive voice or filler. “The details of the pin can be seen by tapping on it.”

              

            


            

              

                
Do

                
Be specific with error messages. “Location access is off. Turn it on in Settings to see nearby places.”

              

              

                
Don't

                
Avoid vague or blaming language. “Something went wrong. Please try again later.”

              

            


            

              

                
Do

                
Describe conditions with neutral language. “This section has uneven lighting.” “Protected bike lane for 400m.”

              

              

                
Don't

                
Never frame conditions as warnings or judgements. “Unsafe street.” “Avoid this area.” “High crime area.”

              

            


            

              

                
Do

                
Use factual comparison in route views. “Route A has more consistent lighting.”

              

              

                
Don't

                
Never use predictive or emotional language. “You may feel unsafe here.” “Safer route.” “Risk increases here.”

              

            


            

              

                

                  
                

                

                  
Capitalization

                  
Use sentence case everywhere — buttons, headings, labels, tabs. Title Case only for the product name “Livemap”.

                

              

              

                

                  
                

                

                  
Language, not colours

                  
Conditions are described in words, not encoded in alarm colours. “Well-lit” is a description. Green-to-red is a judgement.

                

              

            

          


          
          

            
Dark Mode

            
All Livemap surfaces support automatic dark mode via semantic tokens. Never hard-code color values.


            

              

                

                  
                

                

                  
Use semantic color tokens

                  
Always reference --color-surface, --color-label, --color-fill, etc. These swap automatically between light and dark.

                

              

              

                

                  
                

                

                  
Test both modes

                  
Verify every screen in both light and dark. Pay attention to shadows — they need higher opacity in dark mode for visibility.

                

              

              

                

                  
                

                

                  
Elevation shifts in dark mode

                  
In dark mode, elevated surfaces use lighter backgrounds (--color-surface-secondary) rather than relying on shadows alone.

                

              

            

          


          
          

            
Internationalization

            
Livemap is built for a global audience. Design for text expansion, RTL layouts, and locale-specific formatting.


            

              

                

                  
                

                

                  
Allow 40% text expansion

                  
German and other languages can be 30–40% longer than English. Design layouts with flexible widths and avoid fixed-width text containers.

                

              

              

                

                  
                

                

                  
Support RTL layouts

                  
Use logical CSS properties (margin-inline-start, padding-inline-end) instead of physical directions. Mirror icon directions for RTL scripts.

                

              

              

                

                  
                

                

                  
Locale-aware formatting

                  
Use the system locale for dates, times, numbers, and units. Never hardcode formats like "MM/DD/YYYY" — use Intl APIs or platform formatters.
